# Стабильные ссылки для медиа

## Что было сделано и зачем

В проекте UI-Kit используется как внутренняя медиатека: через него загружаются изображения, а затем их URL вставляются в вёрстку страниц. Изначально система копировала прямой путь к физическому файлу в `/public/uploads/`, например:

- `/uploads/deba3c4d-b5c3-46f5-91df-85858826baeb.png`
- `/uploads/eec9c920-dfc5-40f8-bb51-80a8471a0fec.png`

Проблема такого подхода в том, что при замене файла меняется `filename`, а значит меняется и URL. В результате замена изображения в UI-Kit не выполняла главную задачу: старую ссылку в коде всё равно приходилось менять вручную.

Чтобы это исправить, была введена схема стабильных публичных ссылок, завязанных не на имя файла, а на идентификатор записи в таблице `Media`.

Теперь для вставки в вёрстку используется URL вида:

- `/media/12`
- `/media/37`

Где число — это `Media.id`.

## Ключевое решение

Физические файлы по-прежнему хранятся в `/public/uploads/` под UUID-именами. Это удобно для хранения и замены, и не требует менять текущую файловую модель.

Но наружу теперь отдается не прямой путь к файлу, а стабильный route:

- `GET /media/[id]`

Этот route:

1. Находит запись `Media` по `id`.
2. Смотрит актуальные `filename`, `mimeType`, `size`, `originalName`.
3. Читает текущий файл с диска.
4. Возвращает его с правильным `Content-Type`.

За счет этого ссылка остается одной и той же, даже если:

- заменился сам файл;
- изменилось расширение;
- изменился MIME-тип;
- UUID-файл на диске стал другим.

## Как теперь работает замена файла

Раньше при `PATCH /api/admin/media/[id]` происходило следующее:

1. Новый файл сохранялся в `/uploads/<new-uuid>.<ext>`.
2. Старый файл удалялся.
3. В таблице `Media` обновлялись `filename`, `mimeType`, `size` и `url`.
4. `url` становился новым `/uploads/...`, и старая ссылка ломалась.

Теперь логика другая:

1. Новый файл сохраняется на диск.
2. Старый файл удаляется.
3. В таблице `Media` обновляются `filename`, `mimeType`, `size`.
4. `url` принудительно остается стабильным и соответствует `Media.id`, то есть `/media/<id>`.

Таким образом, запись `Media` становится аналогом “ячейки” или “слота”, а файл внутри этой записи можно менять, не ломая внешний адрес.

## Где это реализовано

### Публичная выдача файла

- `app/media/[id]/route.ts`

Это основной публичный route, который по `Media.id` отдает актуальный файл.

### Построение стабильного URL

- `lib/mediaUrl.ts`

Вынесен helper:

- `buildMediaUrl(id)` → возвращает `/media/<id>`
- `withStableMediaUrl(media)` → подменяет `url` у объекта медиа на стабильный

### Загрузка файлов

- `app/api/admin/media/route.ts`

После создания записи `Media` сервер сразу обновляет `url` на стабильный `/media/<id>`. То есть даже новые загрузки с первого дня получают постоянный публичный URL.

### Замена файлов

- `app/api/admin/media/[id]/route.ts`

При замене файла меняются только файловые метаданные. Внешний URL остается завязанным на `id`.

### UI-Kit галерея

- `app/admin/(auth)/ui-kit/ImageGallery.tsx`

Кнопка `Copy path` теперь копирует стабильный URL `/media/<id>`, а не прямой путь в `/uploads`.

### Публичные зависимости от media.url

- `app/api/cv/route.ts`

Этот route тоже переведен на построение стабильного URL через `Media.id`, чтобы PDF/медиа в meta-данных не зависели от физического имени файла.

## Почему это лучше

### Для верстки

Если в `.tsx` или в данных страницы вставлен URL `/media/12`, он не меняется после замены файла в UI-Kit.

Это снимает главную ручную боль:

- не нужно искать старые `src`;
- не нужно обновлять ссылки в коде после replace;
- не нужно следить, какое у нового файла расширение.

### Для CMS-подхода

Запись в таблице `Media` начинает играть роль постоянной сущности, а не временной ссылки на файл. Это ближе к ожидаемой модели “есть слот изображения, содержимое которого можно заменить”.

## Важные ограничения и gotchas

### 1. Старые ссылки `/uploads/...` не мигрируют автоматически

Новая система не переписывает уже вставленные в код старые пути. Если в старой вёрстке уже стоит `/uploads/uuid.png`, такая ссылка останется нестабильной.

Чтобы получить новое поведение, нужно:

- либо заново скопировать путь из UI-Kit;
- либо отдельно сделать миграцию существующих ссылок на `/media/<id>`.

### 2. Кэширование

Для route `/media/[id]` выставлен `Cache-Control: no-store`.

Это сделано специально, чтобы после замены файла браузер не продолжал показывать старую версию по тому же URL.

Минус:

- такой URL не будет кэшироваться агрессивно CDN/браузером.

Это осознанный обмен: стабильность и актуальность сейчас важнее агрессивного кеша.

Если позже понадобится производительность, можно будет перейти на более тонкую стратегию:

- ETag / Last-Modified;
- version query string;
- ручной cache busting.

### 3. Физическое имя файла все еще UUID

Это нормально. UUID-файл нужен только для хранения на диске. Публичный адрес больше не обязан совпадать с именем файла.

### 4. Набор поддерживаемых MIME-типов не расширялся

Текущая задача решала стабильность URL, а не расширение форматов загрузки.

Сейчас `lib/mediaStorage.ts` по-прежнему ограничивает допустимые MIME-типы. Это значит:

- новая схема выдержит смену расширения/типа у уже поддерживаемых форматов;
- но если нужно загружать новые типы вроде MPEG или видео, это отдельная задача.

## Как это связано с остальной системой

Эта механика связывает между собой сразу несколько уровней проекта:

- UI-Kit как интерфейс загрузки и копирования пути;
- Admin API как слой загрузки и замены файлов;
- Prisma `Media` как постоянную запись о ресурсе;
- публичный route `/media/[id]` как стабильную точку доступа;
- sandbox и обычную вёрстку, куда вставляется `src`.

По сути, раньше проект ссылался на файл. Теперь проект ссылается на медиа-сущность.

Это более устойчивая архитектура для CMS-подхода.

## Что стоит помнить дальше

Если в будущем появятся:

- видео;
- аудио;
- разные размеры одного и того же изображения;
- webp/avif-деривативы;
- кропы и responsive sources,

то именно `Media.id` должен оставаться главным стабильным идентификатором, а все производные форматы уже можно привязывать к нему, а не к имени конкретного файла.

Это хорошая база для дальнейшего развития медиаслоя.
